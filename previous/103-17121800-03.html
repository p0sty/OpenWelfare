Finding and completing the work item

Find work item + Read more ...

Work Item from customer enquiry

In Process Direct, from the Snapshot page:

  • in bottom right hand corner of screen select the More Options () icon
  • from the list, select Manage Receipts

On the Pending Payment (PPR) table, it will show:

  • the date, amount and receipt number for the repayment
  • a work item number, if there has been an error with the repayment updating the advance or further action is required

In Customer First:

  • Select CRN/BP
  • Scroll down and select Activities tab
  • Check for CoC Interaction with work item number from PPR
  • Check for status of For Manual Action. If other status, work item has already been finalised
  • In Notes tab, select the note activity
  • Identify the action user/customer was attempting
  • Click on the Edit option
  • In Status field, use drop down list and change For Manual Action to Finalised
  • Click on the Save option
  • Status will change to Completed. If status does not change from For Manual Action, clear record and then repeat process.
  • Finalise the work item prior to any manual action. This allows the updates to auto process in the background and in some cases fix the issue
  • Check the record and action manually if required

If after the work item is finalised the status updates to For Manual Action, complete all manual updates then Complete the work item.

Work Item from Workload manager

  • Enter claim id number in the Search field
  • From Item found list, select the claim id number link
  • In Notes tab, select the note activity
  • Identify the action user/customer was attempting
  • Click on the Edit option
  • In Status field, use drop down list and change For Manual Action to Finalised
  • Click on the Save option
  • Status will change to Completed. If status does not change from For Manual Action, clear record and then repeat process.
  • Finalise the work item prior to any manual action. This allows the updates to auto process in the background and in some cases fix the issue
  • Check the record and action manually if required

If after the work item is finalised the status updates to For Manual Action, complete all manual updates then Complete the work item.

Work item status must be changed from For Manual Action to Finalised for system updates to occur. Any other status will not trigger updates.

See Workload Management for more information on actioning work items.

Draft work items

Draft items + Read more ...

Work items that have a status of Draft and contain details of why the work item was created. In most cases, the system will then complete the action required.

If the draft work item updates to For Manual Action, further actions are required as detailed below.

If the draft work item does not contain any information, do not complete/finalise it.

ZIAM_AVC_GEN

Advance Payment General + Read more ...

This work item is created when an advance payment activity has been started but not completed. This could be due to:

  • customer applying using self -service channels but advance activity could not be finalised
  • Service Officer processing advance request but not finalising activity. For example, activity goes to QOL but is not approved on same day
  • other advance payment activity that needs actioning

Check the Activity List (AL) screen to identify the activity reason. For example:

  • Advance Payment Request
  • Review Reason: End Reduced Recovery

If activity is Advance Payment Request:

  • check the Advance Payment workflow/Manage Advances to see if the advance payment is showing with same date as activity or a date after
  • If current advance:
    • has a status of provisional (PVL), activity needs to be selected and finalised to issue advance. Contact customer before finalising to ensure customer still requires advance payment
    • does not have status of PVL and activity is an Advance Payment Request , cancel the activity

If activity is Review Reason: End Reduced Recovery, the customerâ€™s rate of recovery for the advance has been reduced or suspended:

  • Check Document List (DL) screen to ensure customer has not contacted to ask for further suspension/reduction. If not, finalise activity
  • If further reduction has been actioned, cancel activity

See:

  • Advances
  • Advance payment for non-pension customers
  • Advance payments for pension customers
  • Advance payments of Family Tax Benefit (FTB)
  • Advance payments of Farm Household Allowance

ZADV_CLM_ADJR

Adjust repayment + Read more ...

Work item is created when an adjust repayment, discontinue future advance or adjust instalment date update is made and there is a system failure.

If presented with this work item Service Officers will need to investigate the record and action the work item accordingly.

For details on:

  • Adjusting repayments, see Changes to the repayment rate of an advance payment
  • Discontinuing future advances, see Advance payments of Family Tax Benefit (FTB)
  • Adjusting instalment dates, see:
    • the Advance payment by instalment using Process Direct table in Advance payment for non-pension customers, or
    • the Advance payment by instalment using Process Direct table in Advance payments for pension customers

ZADV_CLM_BUPD

Balance update + Read more ...

Work item is created when an adjust balance activity is submitted but there is a system failure.

If presented with this work item, Service Officers will need to investigate and update the advance balance. In the case of a failed balance transfer for a fully paid off advance, a Service Officer would need to be able to correctly restore an advance.

For details on how to adjust advance payment balances see the Advance repayments - Refund of lump sum payment table in Lump sum repayment of advance payment.

SADV_CLM_PAYA

Repayment + Read more ...

This work item is created when a lump sum payment is made but a corresponding advance record is not updated due to a system issue.

Service Officers will need to:

  • check the work item to identify the reason the repayment could not be updated
  • view the receipt in the Manage Receipts/Pending Payment Receipts (PPR) table in Process Direct, to identify the amount paid and the receipt number
  • check the details match with the Recent Lump Sum Payment table within Manage Advances in Process Direct. If advance has not been updated, advance payment will not have been reduced by amount paid

Update the advance balance to show payment has been received.

For more information, see Lump sum repayment of advance payment.

ZADV_CLM_RERE

Refund request + Read more ...

This work item is created when a Service Officer requests a refund of an entry in the PPR.

When a customer requests a refund of a lump sum advance repayment this work item will be generated 5-7 days after a Service Officer processes the 'request Refund' task.

Specialised staff will be able to complete the refund process.

For detailed information see the Advance repayments - Refund of lump sum payment table in Lump sum repayment of advance payment.

ZADV_CLM_RFND

Refund + Read more ...

This work item is created when a Service Officer submits a refund request.

Specialised staff will be able to complete the refund process.

For detailed information see the Advance repayments - Refund of lump sum payment table in Lump sum repayment of advance payment.

ZADV_CLM_AREC

Payment reconciliation + Read more ...

This can only be processed by Service Officers with Pending Payment table authorisation role.

This work item is created when:

  • A payment reversal is received by ESSentials
  • A receipt in the Pending Payment Receipt table has been pending with no ESSentials reconciliation for >= 7 days
  • A payment is received by ESSentials but is not allocated to an advance or debt
  • There is a difference between the Pending Payment Receipt table and ESSentials for the payment amount

To determine the reason, check the notes attached to the work item, it will advise 1 of these 4 reasons:

  • Manual action is required for the following Pending Payment Receipt reversal entry
  • Manual action is required as the following Pending Payment Receipt has not been reconciled within 7 days
  • Manual action is required as the following Pending Payment Receipt is not identified as a debt or advance
  • Manual action is required as there is a difference in the reconciled amount for the following Pending Payment Receipt

ZADV_CLM_AREC

Manual action is required as the following pending payment receipt is not identified as a debt or advance + Read more ...

This can only be processed by Service Officers with Pending Payment table authorisation role.

See the Payment has been received but not applied to the advance payment table in Lump Sum repayment of advance payment.

ZDMS_SA140

Digital Image Work Item - Claim for Advance of Pharmaceutical Allowance + Read more ...

This work item is created when a customer uploads a Claim for Advance of Pharmaceutical Allowance (SA140) form.

  • Check the Activity List (AL) screen to identify the form code on the left hand side of the activity
  • Check the Document List (DL) screen to identify if there was a triggering event that created this activity

See Pharmaceutical Allowance (PhA) advances

ZDMS_SS352

SS352 Application for Advance Payment + Read more ...

This work item is created when a customer uploads an Application for Advance Payment (SS352) form.

  • Check the customer record to see if the advance payment has already been paid.
  • Action the scan and contact the customer to discuss the Advance payment request per scan, and
  • Check the Activity List (AL) screen to identify the form code on the left hand side of the activity
  • When actioned and completed:
    • Select the Transaction ID from the AL screen
    • Select Edit
    • Change Status from Document Added to Completed

See:

  • Advances
  • Advance payment for non-pension customers
  • Advance payments for pension customers
  • Advance payments of Family Tax Benefit (FTB)
  • Advance payments of Farm Household Allowance

E058AV

Variation, recovery amount or second instalment date must be entered + Read more ...

This issue occurs when a customer is trying to make a payment, update fortnightly recovery amount or change the date of the second instalment.

Update fortnightly recovery amount

As the new recovery amount is not being sent to the mainframe and the information may not be captured in the transaction notes, the Service Officer may not be able to determine the new recovery amount. Future work items will need to be checked to see if an adjust repayment transaction has been made that shows the new amount.

In Customer First:

  • Select CRN/BP
  • Scroll down and select Activities tab
  • Check if CoC Interaction has status of For Manual Action

E054AV

Balance variation exceeds outstanding advance balance + Read more ...

This occurs where the customer pays more than the outstanding balance of the advance usually due to 'rounding of cents'.

  • Select CRN/BP
  • Scroll down and select Activities tab
  • Check if CoC Interaction has status of For Manual Action
  • Select the work item number link
  • Take note of the information within the Note tab
  • Click on the Edit option
  • In Status field, use drop down list and change For Manual Action to Cancelled
  • Click on the Save option
  • Status will change to Completed. If status does not change from For Manual Action, clear record and then repeat process If after the work item is finalised the status updates to For Manual Action, complete all manual updates then Complete the work item
  • In Process Direct, manually update the advance balance

Manual Action to update balance:

  • In Process Direct, in Manage Advances
  • Click on the twisty next to Recent Lump Sum Payments to see all the receipts
  • Look for the coinciding receipt id
  • Take note of the start date and advance balance to identify the advance payment and status
  • Check Current Advance and Previous Advance table to identify the advance
  • Click on the twisty next to the correct advance to see the transaction history
  • If the advance is still current and receipt has not been applied to the advance, the balance will need adjusting by the amount that has been paid. See the Payment has been received but not applied to the advance payment table in Lump Sum repayment of advance payment
  • Once the advance has already been finalised, the over-recovered amount will need to be repaid to the customer via PYAJ if the amount is over $1. If less than $1, update balance of another advance of same type (for example, FTB advance to FTB advance)
